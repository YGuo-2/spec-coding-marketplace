# Spec Coding 完整使用示例

本文档展示了一个端到端的 Spec-Driven Development 工作流示例，
演示如何使用 spec-coding 技能从需求到代码的完整流程。

---

## 场景：为博客应用添加评论功能

### 状态 A：需求审视与澄清

**用户输入：**
> 我需要给博客应用添加一个评论功能，用户可以在文章下面发表评论。

**智能体响应：**

## 📋 需求澄清清单

在进入规范编写阶段之前，我需要确认以下问题：

1. 评论是否支持**嵌套回复**（即评论的评论），还是仅支持平级评论？
2. 评论是否需要**登录后**才能发表？匿名评论是否被允许？
3. 是否需要**评论审核机制**（如管理员审批后才显示）？
4. 评论是否支持 **Markdown 格式** 或仅支持纯文本？
5. 单条评论是否有**字数限制**？如果有，上限是多少？
6. 是否需要**评论编辑/删除**功能？谁有权限？
7. 是否需要**反垃圾/敏感词过滤**机制？
8. 现有技术栈是什么？（前端框架、后端语言、数据库类型）

**用户回复：**
> 1. 仅平级评论，不需要嵌套
> 2. 必须登录，不允许匿名
> 3. 暂时不需要审核
> 4. 纯文本即可
> 5. 最多 500 字
> 6. 作者可以删除自己的评论，管理员可删除任何评论
> 7. 暂不需要
> 8. React + Node.js + Express + PostgreSQL

---

### 状态 B：规范工件生成

智能体生成以下三个文件：

#### docs/specs/product.md（节选）

```markdown
## 2. 用户故事与验收标准

### US-001: 发表评论

**作为**已登录用户，**我希望**能在文章下方发表评论，**以便**分享我的观点。

#### 验收标准

- **AC-001.1:**
  - **GIVEN** 用户已登录且正在查看一篇文章
  - **WHEN** 用户在评论框中输入文字（1-500字）并点击"发布"
  - **THEN** 评论立即显示在评论列表中，包含用户名和时间戳

- **AC-001.2:**
  - **GIVEN** 用户未登录
  - **WHEN** 用户尝试发表评论
  - **THEN** 系统提示"请先登录"并提供登录链接
```

#### docs/specs/architecture.md（节选）

```markdown
## 3. 数据模型

### comments 表

| 字段名 | 类型 | 约束 | 描述 |
|:---|:---|:---|:---|
| id | UUID | PK | 评论主键 |
| post_id | UUID | FK → posts.id, NOT NULL | 所属文章 |
| author_id | UUID | FK → users.id, NOT NULL | 评论作者 |
| content | TEXT | NOT NULL, maxlength=500 | 评论内容 |
| created_at | TIMESTAMP | NOT NULL, DEFAULT NOW | 创建时间 |
```

#### docs/specs/tasks.md（节选）

```markdown
## 阶段 1：基础设施

- [ ] **T-001:** 创建 comments 数据库迁移文件
  - 📁 涉及文件: `migrations/003_create_comments.sql`
  - ✅ 验证标准: 迁移可正常执行，comments 表创建成功
  - ⏱️ 预估: 30 分钟
  - 🔗 依赖: 无

## 阶段 2：核心逻辑

- [ ] **T-002:** 实现 Comment 数据模型和 Repository
  - 📁 涉及文件: `src/models/Comment.js`, `src/repositories/commentRepository.js`
  - ✅ 验证标准: 单元测试覆盖 CRUD 操作
  - ⏱️ 预估: 1 小时
  - 🔗 依赖: T-001

- [ ] **T-003:** 实现评论 Service 层（含权限验证）
  - 📁 涉及文件: `src/services/commentService.js`
  - ✅ 验证标准: 仅作者和管理员可删除评论的逻辑测试通过
  - ⏱️ 预估: 1 小时
  - 🔗 依赖: T-002
```

**智能体输出：**
> ✅ 规范工件生成完毕。请审查 `docs/specs/` 下的三个文件。
> 确认无误后，请回复 **"批准规范，启动执行"**。

**用户回复：**
> 批准规范，启动执行

## 关键要点总结

1. **绝不跳过状态 A** — 即使需求看似简单，也要主动发问
2. **文档先于代码** — 三个规范文件是"可执行的工件"
3. **逐任务推进** — 严禁贪多，一次只做一个复选框
4. **自愈合循环** — 测试失败时自动修复，最多 3 次
5. **退回机制** — 发现架构缺陷时立即回退修改规范
