# Bugfix Spec 完整使用示例

本文档展示了一个端到端的 Bugfix Spec 工作流示例，
演示如何用 spec-coding 技能处理“先复现、再修复、最后防回归”的缺陷修复任务。

---

## 场景：修复订单接口重复提交导致双扣库存

### 路由判断

**用户输入：**
> 线上有个 bug，订单接口偶尔会重复扣减库存。帮我修一下。

**智能体响应：**

我读到了Spec-coding技能。

## Spec 路由决定

- 路径：Bugfix
- Design-First 粒度：n/a
- 原因：目标是修复既有错误行为，并且涉及库存一致性与线上回归风险
- 下一步：Bug 分析澄清

我会按照“Bugfix”分支来完成。

> [!WARNING]
> 高风险变更警告：当前任务涉及生产库存一致性与线上回归风险，必须进行人类深度审查，切勿草率合并。

---

### 状态 A：Bug 分析与澄清

**智能体响应：**

## 🐛 Bug 修复澄清清单

在进入 bugfix 规范阶段之前，我需要确认以下问题：

1. 重复扣库存时，是否伴随重复创建订单，还是只有库存被重复扣减？
2. 该问题出现在哪些环境和版本？是否能稳定复现？
3. 是否已有错误日志、Trace ID、失败请求样本或最近相关改动？
4. 修复后哪些行为必须保持不变？例如正常下单性能、幂等键策略、库存扣减时机。
5. 目前是否已有失败测试或监控告警可以作为证据？

**用户回复：**
> 1. 订单只会创建一次，但库存有时会扣两次  
> 2. 只在生产出现，预发没复现  
> 3. 有两条 Trace，最近上周把下单链路改成了异步重试  
> 4. 不能改变现有接口契约，也不能影响正常下单性能  
> 5. 目前没有自动化失败测试

---

### 状态 B：Bugfix 规范工件生成

智能体生成以下三个文件：

#### docs/specs/bugfix.md（节选）

```markdown
## 3. 当前错误行为 (Current Behavior)

### BUG-001: 重试请求导致库存重复扣减

- **WHEN** 同一订单请求在异步重试窗口内被重复消费
- **THEN** 系统当前会再次调用库存扣减逻辑，导致库存被重复扣减

## 5. 必须保持不变的行为 (Unchanged Behavior)

### SAFE-001: 正常下单链路保持原有响应契约

- **WHEN** 用户提交一次合法下单请求
- **THEN** 系统仍应保持现有响应结构、状态码和性能目标不变
```

#### docs/specs/design.md（节选）

```markdown
## 1. 根因分析

- 根因：异步重试后的消息消费路径缺少以订单号为粒度的幂等保护

## 4. 测试与验证策略

### 4.1 复现证明

- 新增集成测试，模拟同一订单消息被消费两次

### 4.3 回归防护

- 保留正常单次下单流程测试，确认接口响应契约与性能基线不变
```

#### docs/specs/tasks.md（节选）

```markdown
- [ ] **B-001:** 建立重复消费导致双扣库存的失败证明
  - ✅ 验证标准：未修复前集成测试稳定失败，并能证明库存扣减执行了两次

- [ ] **B-002:** 为库存扣减路径添加订单粒度幂等保护
  - ✅ 验证标准：B-001 转为通过，且不改变现有下单接口契约

- [ ] **B-003:** 补充正常单次下单与并发相邻路径回归测试
  - ✅ 验证标准：证明修复未影响正常下单流程
```

**智能体输出：**
> ✅ Bugfix 规范工件生成完毕。请审查 `docs/specs/bugfix.md`、`docs/specs/design.md` 和 `docs/specs/tasks.md`。  
> 确认无误后，请回复 **"批准 bugfix 规范，启动执行"**。

**用户回复：**
> 批准 bugfix 规范，启动执行

---

### 状态 C：受控代码实施

**智能体：** 📌 锁定任务 **B-001**：建立重复消费导致双扣库存的失败证明

*(智能体先编写失败证明，再继续后续任务。不会直接跳到修复代码。)*

**验证通过后：**
> ✅ B-001 完成。tasks.md 已更新为 `[x]`。  
>
> 建议 commit message:
> ```
> fix(order): capture duplicate inventory deduction regression
>
> Implements task: B-001 - establish failing proof for duplicate inventory deduction
> Spec: docs/specs/tasks.md
> ```
>
> 是否继续执行 **B-002**？

---

## 关键要点总结

1. **先确认这是 Bugfix，不是伪装的新功能**
2. **先写清当前错误行为、期望行为和不变行为**
3. **先建立失败证明，再做最小修复**
4. **回归防护是必需品，不是可选项**
5. **如果修复本质上变成了产品能力变更，就要切回 Feature Spec**
